To design a non-blocking search infrastructure for your SQLite-backed crawler, we must balance SQLite's **Write-Ahead Logging (WAL) mode** (which allows concurrent readers and a single writer) with efficient query execution.

### Step-by-Step Thought Process

1.  **Non-Blocking Logic**: Since `aiosqlite` and SQLite's **WAL mode** are already integrated, readers (search) do not block the writer (crawler). We will ensure the `search.py` connection uses `PRAGMA query_only = ON` to enforce read-only safety, which SQLite optimizes.
2.  **Query Structure**: Rather than pulling all rows into memory and filtering in Python, we will use SQL `LIKE` operators with indices. To handle the `(relevant_url, origin_url, depth)` return format efficiently, we will define a covering index.
3.  **Performance Optimization**: 
    *   **Full-Text Search (FTS5)**: Standard `LIKE` searches are slow on large datasets. We will implement `FTS5`, which builds an inverted index, making searches near-instant.
    *   **Pagination**: Instead of returning all results, we'll add `LIMIT` and `OFFSET`.
4.  **Integration**: We create a dedicated `search.py` that encapsulates the high-speed FTS5 query logic.

### 2. Code Generation: `search.py`

This module is designed to be imported by your `api.py` and `main.py`, replacing the slow Python-side filtering.

```python
import sqlite3
from typing import List, Dict

DB_PATH = "crawler.db"

def search(query: str, limit: int = 20, offset: int = 0) -> List[Dict]:
    """
    Optimized search using FTS5 and ranked scoring.
    Returns: list of dicts with (url, origin_url, depth, score)
    """
    conn = sqlite3.connect(DB_PATH)
    # Enable WAL mode for concurrency
    conn.execute("PRAGMA journal_mode=WAL;")
    
    # We use FTS5 ranking (bm25) for relevance
    # We join with Pages to get metadata (origin_url, depth)
    sql = """
    SELECT p.url, p.origin_url, p.depth, 
           bm25(s, 10.0, 5.0, 1.0) AS score
    FROM Pages_Search s
    JOIN Pages p ON s.url = p.url
    WHERE Pages_Search MATCH ?
    ORDER BY score ASC
    LIMIT ? OFFSET ?
    """
    
    # FTS5 syntax: 'title: query OR body: query'
    fts_query = f'"{query}" OR title:"{query}" OR body:"{query}"'
    
    cursor = conn.cursor()
    cursor.execute(sql, (fts_query, limit, offset))
    
    results = [
        {"url": row[0], "origin_url": row[1], "depth": row[2], "score": round(abs(row[3]), 2)}
        for row in cursor.fetchall()
    ]
    
    conn.close()
    return results
```

```python
    # 2. Update/Insert FTS5 virtual table
    await self.db.execute("""
        INSERT OR REPLACE INTO Pages_Search(url, title, body)

    """, (url, title, body))

```

### Key Performance Benefits:
*   **Concurrency**: By using `WAL` mode, the crawler can perform massive writes while the API simultaneously executes `SELECT` queries without "Database is locked" errors.
*   **Indexing**: The `Pages_Search` FTS5 table is an inverted index (like a book's index). It avoids a full table scan, meaning even if your database reaches 1,000,000 pages, the search time will remain in milliseconds.
*   **BM25 Ranking**: The `bm25` function is the industry standard for search relevance (better than simple frequency counts), providing professional-grade results out of the box.

### 1. Chain of Thought: Strategy

1.  **Concurrency Model:** By setting `PRAGMA journal_mode=WAL`, SQLite maintains a `-wal` file. Readers access the database state from a snapshot without waiting for the writer to finish committing its transaction to the main file.
2.  **Indexing vs. Searching:** We will use standard SQL `LIKE` queries for basic keyword matching. To make this performant without external dependencies (like FTS5 if not strictly necessary, though FTS5 is highly recommended for performance), we index the `title` and `body` columns.
3.  **Search Logic:** Since we need `(relevant_url, origin_url, depth)`, we will return a structured list. The scoring algorithm provided in your example is perfect, as it keeps the computation inside the Python process, keeping the SQL query lean.
4.  **Performance:** We will implement an optional **FTS5 (Full-Text Search)** table if the dataset grows, but for a "mini" engine, standard indexing on columns combined with `LIMIT` and `OFFSET` (for pagination) is sufficient.

### 2. Implementation: `search.py`

This module is designed to be imported by your `api.py` and `main.py`.

```python
import sqlite3
from database import DB_PATH, turkish_lower

def search(query: str, limit: int = 20):
    """
    Non-blocking search implementation.
    The 'query' is performed against the Pages table.
    We return the top N results based on simple keyword frequency.
    """
    query_lower = turkish_lower(query)
    
    # 1. Connect with read-only access where possible
    conn = sqlite3.connect(DB_PATH)
    conn.execute("PRAGMA journal_mode=WAL;")
    
    # 2. Querying only columns we need to minimize I/O
    # We use a LIKE clause to let SQLite filter out rows that don't match at all
    # This is much faster than pulling the entire database into memory.
    sql = """
        SELECT url, origin_url, depth, title, body 
        FROM Pages 
        WHERE title LIKE ? OR body LIKE ?
    """
    
    pattern = f"%{query_lower}%"
    cursor = conn.execute(sql, (pattern, pattern))
    rows = cursor.fetchall()
    conn.close()

    # 3. Calculate scores in-memory (lightweight)
    results = []
    for url, origin, depth, title, body in rows:
        title_l = turkish_lower(title or "")
        body_l = turkish_lower(body or "")
        
        freq = title_l.count(query_lower) + body_l.count(query_lower)
        
        if freq > 0:
            # Score formula: (frequency * 10) + 1000 - (depth * 5)
            score = (freq * 10) + 1000 - (depth * 5)
            results.append({
                "url": url,
                "origin_url": origin,
                "depth": depth,
                "score": score
            })

    # 4. Return top results sorted by score
    return sorted(results, key=lambda x: x['score'], reverse=True)[:limit]
```

### 3. Optimization Checklist

*   **WAL Mode (Vital):** In `database.py`, ensure `init_db` always runs `PRAGMA journal_mode=WAL;`. This is the #1 requirement for non-blocking concurrent read/writes.
*   **Indexing:** Ensure you have added these indexes to `database.py` to speed up the `WHERE` clause:
    ```python
    conn.execute("CREATE INDEX IF NOT EXISTS idx_pages_title ON Pages(title);")
    conn.execute("CREATE INDEX IF NOT EXISTS idx_pages_body ON Pages(body);")
    ```
*   **Memory Management:** If the `Pages` table grows into the hundreds of thousands, fetching `*` will be slow. The `search.py` implementation above only fetches necessary columns and uses SQL-side filtering (`LIKE`) to ensure only relevant rows are passed to the scoring algorithm.
*   **FTS5 (Pro-tip):** If you find `LIKE` queries getting slow, modify your `init_db` to include a Virtual Table:
    ```sql
    CREATE VIRTUAL TABLE Pages_FTS USING fts5(url, title, body);
    ```
    This turns search into a sub-millisecond operation regardless of dataset size, as it creates an inverted index automatically. 

This infrastructure satisfies your requirements by separating the concerns of the crawler (writing) and the searcher (reading) while utilizing SQLite’s robust built-in concurrency features.

### 1. Design Thought Process

*   **Concurrency & WAL Mode:** Since you have enabled `PRAGMA journal_mode=WAL;`, the database will not block readers during writes. We will leverage this by using a separate `sqlite3` connection for the search module, distinct from the crawler's connection.
*   **Search Logic:** Instead of complex external engines, we will use SQL's `LIKE` operator with potential for a `FTS5` (Full-Text Search) migration later. To keep it simple and effective, we will perform a rank-based query that prioritizes matches in the `title` over the `body`.
*   **Non-blocking Execution:** The search will be implemented as an `async` function using `run_in_executor` to prevent the heavy CPU-bound string matching from blocking the main event loop, keeping the CLI responsive.

### 2. Search Infrastructure (`search.py`)

```python
import sqlite3
import asyncio
from concurrent.futures import ThreadPoolExecutor

class SearchEngine:
    def __init__(self, db_path="crawler.db"):
        self.db_path = db_path
        self.executor = ThreadPoolExecutor(max_workers=3)

    def _execute_search(self, query):
        """
        Synchronous search logic executed in a thread pool.
        Matches keywords against title and body.
        """
        conn = sqlite3.connect(self.db_path)
        conn.row_factory = sqlite3.Row
        cursor = conn.cursor()

        # Simple rank-based SQL: 
        # Title matches get higher priority.
        # We search for any part of the term using %term%.
        sql = """
        SELECT url, origin_url, depth, title,
               ( (title LIKE ?) * 10 + (body LIKE ?) * 1 ) as rank
        FROM Pages 
        WHERE title LIKE ? OR body LIKE ?
        ORDER BY rank DESC
        LIMIT 20
        """
        
        search_term = f"%{query}%"
        cursor.execute(sql, (search_term, search_term, search_term, search_term))
        results = cursor.fetchall()
        
        conn.close()
        return [dict(row) for row in results]

    async def search(self, query):
        """
        Async wrapper to provide non-blocking search execution.
        """
        loop = asyncio.get_event_loop()
        results = await loop.run_in_executor(self.executor, self._execute_search, query)
        return results

# results = await engine.search("python")
```

### 3. Performance Optimizations for SQLite

To ensure the search remains fast as your `Pages` database grows:

1.  **Indexing:** Even if you use `LIKE`, ensure you have an index on the `title` column.
    *   *Action:* Run this command once in your DB setup: `CREATE INDEX idx_pages_title ON Pages(title);`
2.  **Projection:** We only fetch necessary columns (`url`, `origin_url`, `depth`). Avoid `SELECT *`.
3.  **FTS5 Migration (Future-proofing):** If the database exceeds 100,000 pages, the `LIKE` operator will slow down significantly. I recommend creating an FTS5 virtual table that shadows the `Pages` table. It will automatically keep the index updated and perform lightning-fast prefix or boolean searches.
4.  **Memory Management:** The use of `ThreadPoolExecutor` ensures that even if a search takes 500ms to scan the disk, your `Crawler` workers continue downloading pages on the main thread without interruption.

### 4. Why this fits your Architecture:
*   **No Locks:** By using a local connection inside the thread, we rely on SQLite’s built-in WAL reader-writer concurrency.
*   **Simple:** It requires no external dependencies, sticking strictly to standard library features.
*   **Scalable:** By offloading to an `executor`, we ensure that the search engine feels instantaneous to the user, even while the crawler is working at full capacity in the background.
