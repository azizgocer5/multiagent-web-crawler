To design a robust SQLite infrastructure for a high-concurrency crawler/search engine, we must address the **"Single Writer, Multiple Reader"** bottleneck inherent to SQLite.

### 1. Architectural Strategy
*   **WAL Mode (Write-Ahead Logging):** Essential for concurrency. It allows multiple readers to access the database while a writer is modifying it, significantly reducing lock contention.
*   **Synchronous = NORMAL:** In WAL mode, `NORMAL` is safe and significantly faster than `FULL` because the WAL file manages the integrity, allowing the OS to handle the actual disk syncs.
*   **Busy Timeout:** We set a `busy_timeout` (e.g., 5000ms) to allow threads to wait gracefully if the database is locked, rather than throwing immediate operational errors.
*   **Schema Optimization:** We use `WITHOUT ROWID` or standard indexes to ensure that URL lookup (for deduplication) and state checking remain $O(\log n)$.

### 2. Implementation (`database.py`)

### 3. Key Design Considerations for Concurrency
1.  **Shared vs. Individual Connections:** 
    *   **Crawler:** Use a single persistent `aiosqlite` connection per background worker/process to avoid the overhead of re-opening the file.
    *   **API/Search:** Open a new connection per request or use a small connection pool.
2.  **Transaction Granularity:** In `database.py`, keep transactions short. For bulk inserts of found links during a crawl, use `executemany` inside a single `COMMIT` block. This is the single most effective way to increase write throughput.
3.  **State Management:** By separating `Queue` (ephemeral) from `Pages` (durable), we ensure that even if the crawl is interrupted, the search engine remains functional and fast.
4.  **Why no ORM?** ORMs (SQLAlchemy, Peewee) add overhead and abstraction layers that hide the raw SQLite performance tuning flags (like `journal_mode`) required for high-concurrency crawling. Using `aiosqlite` directly provides maximum control.

```python
    """Synchronous init for main thread bootstrapping."""
    conn = get_connection(async_mode=False)
    # Re-use logic defined in init_db via execution block
    conn.executescript("""
        CREATE TABLE IF NOT EXISTS Pages (id INTEGER PRIMARY KEY AUTOINCREMENT, url TEXT UNIQUE, origin_url TEXT, depth INTEGER, title TEXT, body TEXT);
        CREATE TABLE IF NOT EXISTS Queue (id INTEGER PRIMARY KEY AUTOINCREMENT, url TEXT UNIQUE, origin_url TEXT, depth INTEGER, state TEXT DEFAULT 'pending');
        CREATE TABLE IF NOT EXISTS Settings (key TEXT PRIMARY KEY, value TEXT);
        CREATE INDEX IF NOT EXISTS idx_queue_state ON Queue(state);
    """)
    conn.commit()
    conn.close()

def turkish_lower(text: str) -> str:
    """Handles Turkish character casing for accurate search."""
    if not text: return ""
    return text.replace('İ', 'i').replace('I', 'ı').lower()

def search(query: str):
    """
    Performs search. 
    Optimized: In a production scale, use FTS5 extension here.
    For this mini-engine, we perform memory-based scoring.
    """
    conn = get_connection(async_mode=False)
    cur = conn.execute("SELECT url, origin_url, depth, title, body FROM Pages")
    rows = cur.fetchall()
    conn.close()

    query_lower = turkish_lower(query)
    results = []
    
    for url, origin_url, depth, title, body in rows:
        title_norm = turkish_lower(title or "")
        body_norm = turkish_lower(body or "")
        
        freq = title_norm.count(query_lower) * 3 + body_norm.count(query_lower)
        if freq > 0:
            results.append({
                "url": url,
                "origin_url": origin_url,
                "depth": depth,
                "score": freq + 1000 - (depth * 5),
                "title": title
            })
            
    return sorted(results, key=lambda x: x["score"], reverse=True)

def get_stats():
    """Returns database metadata for monitoring."""
    conn = get_connection(async_mode=False)
    try:
        pages = conn.execute("SELECT COUNT(*) FROM Pages").fetchone()[0]
        pending = conn.execute("SELECT COUNT(*) FROM Queue WHERE state='pending'").fetchone()[0]
        proc = conn.execute("SELECT COUNT(*) FROM Queue WHERE state='processing'").fetchone()[0]
        return pages, pending, proc, "Aktif" if pending > MAX_QUEUE_NORMAL else "Normal"
    finally:
        conn.close()

def set_setting(key: str, value: str):
    conn = get_connection(async_mode=False)
    conn.execute("INSERT OR REPLACE INTO Settings (key, value) VALUES (?, ?)", (key, value))
    conn.commit()
    conn.close()

def get_setting(key: str):
    conn = get_connection(async_mode=False)
    row = conn.execute("SELECT value FROM Settings WHERE key = ?", (key,)).fetchone()
    conn.close()
    return row[0] if row else None
```

1.  **WAL Mode**: By setting `PRAGMA journal_mode=WAL`, you allow multiple readers (search API) and one writer (the crawler) to operate simultaneously without locking each other out.
2.  **Indexing**: The index `idx_queue_state` is critical. Without it, the crawler would perform a full table scan of the `Queue` every time it tries to find `pending` URLs, which becomes exponentially slower as the database grows.
3.  **Scalability**: If you decide to expand the `Pages` table to millions of rows, consider replacing the `search` function's Python loop with `sqlite3`'s native **FTS5 (Full Text Search)** module, which is built into almost all modern SQLite distributions.

### `database.py`

```python
import sqlite3
import asyncio
from typing import List, Tuple, Dict, Any

class Database:
    def __init__(self, db_path: str = "crawler.db"):
        self.db_path = db_path
        self.lock = asyncio.Lock()  # Protects concurrent write access
        self._initialize_db()

    def _get_connection(self):
        # Always return a connection with row_factory for dict-like access
        conn = sqlite3.connect(self.db_path)
        conn.row_factory = sqlite3.Row
        return conn

    def _initialize_db(self):
        """Sets up tables and WAL mode for concurrency."""
        with self._get_connection() as conn:
            conn.execute("PRAGMA journal_mode=WAL;")  # Enable concurrent read/write
            conn.execute("PRAGMA synchronous=NORMAL;")
            
            # Pages table
            conn.execute("""
                CREATE TABLE IF NOT EXISTS pages (
                    id INTEGER PRIMARY KEY AUTOINCREMENT,
                    url TEXT UNIQUE,
                    origin_url TEXT,
                    depth INTEGER,
                    title TEXT,
                    body TEXT
                )
            """)
            
            # Queue table
            conn.execute("""
                CREATE TABLE IF NOT EXISTS queue (
                    id INTEGER PRIMARY KEY AUTOINCREMENT,
                    url TEXT UNIQUE,
                    state TEXT DEFAULT 'pending', 
                    depth INTEGER
                )
            """)
            
            # Settings table
            conn.execute("""
                CREATE TABLE IF NOT EXISTS settings (
                    key TEXT PRIMARY KEY,
                    value TEXT
                )
            """)
            conn.commit()

    async def execute_batch(self, queries: List[Tuple[str, tuple]]):
        """Executes multiple writes safely under a lock."""
        async with self.lock:
            conn = self._get_connection()
            try:
                for sql, params in queries:
                    conn.execute(sql, params)
                conn.commit()
            finally:
                conn.close()

    async def get_pending_urls(self, limit: int = 100) -> List[Dict]:
        """Fetch pending URLs and mark them as 'processing'."""
        async with self.lock:
            conn = self._get_connection()
            cursor = conn.cursor()
            
            # Atomic update + select to prevent race conditions
            cursor.execute("""
                UPDATE queue SET state = 'processing' 
                WHERE id IN (SELECT id FROM queue WHERE state = 'pending' LIMIT ?)
                RETURNING url, depth
            """, (limit,))
            
            results = [dict(row) for row in cursor.fetchall()]
            conn.commit()
            conn.close()
            return results

    async def save_page_and_links(self, page_data: Dict, new_links: List[Tuple[str, int]]):
        """Saves scraped content and new links in one transaction."""
        async with self.lock:
            conn = self._get_connection()
            try:
                # 1. Save content
                conn.execute("""
                    INSERT OR REPLACE INTO pages (url, origin_url, depth, title, body)
                    VALUES (?, ?, ?, ?, ?)
                """, (page_data['url'], page_data['origin_url'], page_data['depth'], 
                      page_data['title'], page_data['body']))
                
                # 2. Mark current as done
                conn.execute("UPDATE queue SET state = 'done' WHERE url = ?", (page_data['url'],))
                
                # 3. Add new links (ignore duplicates)
                conn.executemany("""
                    INSERT OR IGNORE INTO queue (url, state, depth) VALUES (?, 'pending', ?)
                """, new_links)
                
                conn.commit()
            finally:
                conn.close()

    async def search(self, query: str) -> List[Dict]:
        """
        Search engine: Fetch pages and perform ranking.
        Since it's a read operation, no explicit lock is required due to WAL mode.
        """
        conn = self._get_connection()
        # Fetching all to perform scoring in memory (as requested by logic)
        cursor = conn.execute("SELECT * FROM pages")
        rows = [dict(row) for row in cursor.fetchall()]
        conn.close()
        
        return rows # Ranking logic should be applied on this result set
```

### Key Technical Decisions:

1.  **WAL Mode:** Enabled `PRAGMA journal_mode=WAL`. This is the gold standard for SQLite concurrency, allowing the `search` function (Reader) to operate without being blocked by `save_page_and_links` (Writer).
2.  **`asyncio.Lock`:** Even with WAL, SQLite can only have one writer at a time. The `self.lock` acts as a semaphore for the worker threads to ensure the `INSERT`/`UPDATE` batch transactions don't collide.
3.  **Atomic State Management:** In `get_pending_urls`, I used the `RETURNING` clause (available in SQLite 3.35+). This atomizes the process of "finding a pending URL" and "marking it as processing," preventing two workers from grabbing the same URL.
4.  **`executemany`:** Used for link insertion to ensure that hundreds of links found on a page are pushed to the database in a single disk I/O operation.
5.  **`INSERT OR IGNORE`:** Handles deduplication at the database level for the `queue` table, automatically ensuring we never crawl the same URL twice.

```python
import sqlite3
import asyncio
from typing import List, Tuple, Dict, Any

class Database:
    def __init__(self, db_path: str = "crawler.db"):
        self.db_path = db_path
        self._lock = asyncio.Lock()
        self._initialize_db()

    def _get_connection(self):
        """Creates a connection with WAL mode enabled for concurrent read/writes."""
        conn = sqlite3.connect(self.db_path)
        conn.execute("PRAGMA journal_mode=WAL;")
        conn.execute("PRAGMA synchronous=NORMAL;")
        conn.row_factory = sqlite3.Row
        return conn

    def _initialize_db(self):
        """Creates tables if they do not exist."""
        with self._get_connection() as conn:
            conn.executescript("""
                CREATE TABLE IF NOT EXISTS Pages (
                    id INTEGER PRIMARY KEY AUTOINCREMENT,
                    url TEXT UNIQUE,
                    origin_url TEXT,
                    depth INTEGER,
                    title TEXT,
                    body TEXT
                );
                CREATE TABLE IF NOT EXISTS Queue (
                    id INTEGER PRIMARY KEY AUTOINCREMENT,
                    url TEXT UNIQUE,
                    state TEXT DEFAULT 'pending',
                    depth INTEGER
                );
                CREATE INDEX IF NOT EXISTS idx_queue_state ON Queue(state);
            """)

    async def execute_batch(self, query: str, params: List[tuple]):
        """Executes a batch operation under an async lock."""
        async with self._lock:
            with self._get_connection() as conn:
                conn.executemany(query, params)
                conn.commit()

    async def get_pending_urls(self, limit: int = 100) -> List[Dict[str, Any]]:
        """Fetches pending URLs and marks them as 'processing'."""
        async with self._lock:
            with self._get_connection() as conn:
                # Select pending
                cursor = conn.execute(
                    "SELECT url, depth FROM Queue WHERE state = 'pending' LIMIT ?", (limit,)
                )
                rows = [dict(row) for row in cursor.fetchall()]
                
                if rows:
                    urls = [r['url'] for r in rows]
                    conn.execute(
                        f"UPDATE Queue SET state = 'processing' WHERE url IN ({','.join(['?']*len(urls))})", 
                        urls
                    )
                    conn.commit()
                return rows

    async def save_page_and_links(self, page_data: Dict, new_links: List[Tuple]):
        """
        Atomically saves a crawled page and queues new discovered links.
        page_data: {'url', 'origin_url', 'depth', 'title', 'body'}
        new_links: List of (url, depth)
        """
        async with self._lock:
            with self._get_connection() as conn:
                try:
                    # 1. Insert/Update Page
                    conn.execute("""
                        INSERT OR REPLACE INTO Pages (url, origin_url, depth, title, body) 
                        VALUES (?, ?, ?, ?, ?)
                    """, (page_data['url'], page_data['origin_url'], page_data['depth'], 
                          page_data['title'], page_data['body']))
                    
                    # 2. Update Queue state to 'done'
                    conn.execute("UPDATE Queue SET state = 'done' WHERE url = ?", (page_data['url'],))
                    
                    # 3. Add new links to queue (ignore duplicates)
                    conn.executemany("""
                        INSERT OR IGNORE INTO Queue (url, depth, state) VALUES (?, ?, 'pending')
                    """, new_links)
                    
                    conn.commit()
                except Exception as e:
                    conn.rollback()
                    raise e

    async def search(self, query: str) -> List[Dict]:
        """
        Performs a search across Pages. 
        Note: Simple logic intended to be used by the SearchEngine.
        """
        async with self._lock:
            with self._get_connection() as conn:
                # We fetch all to perform ranking in Python as per requirements
                cursor = conn.execute("SELECT * FROM Pages")
                return [dict(row) for row in cursor.fetchall()]

# Exported API
__all__ = ['Database', 'execute_batch', 'get_pending_urls', 'save_page_and_links', 'search']
```

### Key Design Decisions:
1.  **Thread/Task Safety:** Even though SQLite handles some concurrency via WAL, `asyncio.Lock` ensures that multiple worker tasks don't attempt to `COMMIT` simultaneously, which can occasionally cause `SQLITE_BUSY` errors during high-frequency writes.
2.  **Atomicity:** `save_page_and_links` is wrapped in a single transaction. This guarantees that we never save a page without updating its state in the queue, preventing "infinite loop" scenarios where the same URL is re-crawled.
3.  **Performance:** By using `executemany` within the `execute_batch` and `save_page_and_links` methods, we minimize the I/O overhead of SQLite disk syncs.
4.  **Decoupling:** The `Database` class is now a pure service. Other modules (`indexer.py`, `search.py`) must import this class and share an instance, ensuring that only one connection manager exists per process.

```python
import asyncio
from typing import List, Dict, Any

class Database:
    def __init__(self, db_path: str = "crawler.db"):
        self.db_path = db_path
        self.write_lock = asyncio.Lock()

    async def initialize(self):
        """Creates tables and enables WAL mode for concurrency."""
        async with aiosqlite.connect(self.db_path) as db:
            await db.execute("PRAGMA journal_mode=WAL;")
            # Pages Table
            await db.execute("""
                CREATE TABLE IF NOT EXISTS pages (
                    id INTEGER PRIMARY KEY AUTOINCREMENT,
                    url TEXT UNIQUE,
                    origin_url TEXT,
                    depth INTEGER,
                    title TEXT,
                    body TEXT
                )
            """)
            # Queue Table
            await db.execute("""
                CREATE TABLE IF NOT EXISTS queue (
                    id INTEGER PRIMARY KEY AUTOINCREMENT,
                    url TEXT UNIQUE,
                    state TEXT DEFAULT 'pending',
                    depth INTEGER
                )
            """)
            # Settings Table
            await db.execute("""
                CREATE TABLE IF NOT EXISTS settings (
                    key TEXT PRIMARY KEY,
                    value TEXT
                )
            """)
            await db.commit()

    async def save_page_and_links(self, page_data: Dict[str, Any], links: List[str]):
        """Atomic write operation for a processed page and new discovery."""
        async with self.write_lock:
            async with aiosqlite.connect(self.db_path) as db:
                try:
                    # 1. Update Page
                    await db.execute("""
                        INSERT OR REPLACE INTO pages (url, origin_url, depth, title, body)
                        VALUES (?, ?, ?, ?, ?)
                    """, (page_data['url'], page_data['origin_url'], page_data['depth'], 
                          page_data['title'], page_data['body']))
                    
                    # 2. Mark queue as done
                    await db.execute("UPDATE queue SET state = 'done' WHERE url = ?", (page_data['url'],))
                    
                    # 3. Add new links (ignoring duplicates)
                    for link in links:
                        await db.execute("""
                            INSERT OR IGNORE INTO queue (url, state, depth)
                            VALUES (?, 'pending', ?)
                        """, (link, page_data['depth'] + 1))
                    
                    await db.commit()
                except Exception as e:
                    await db.rollback()
                    raise e

    async def get_pending_urls(self, limit: int = 10) -> List[Dict]:
        """Fetch URLs to process and mark them as 'processing'."""
        async with self.write_lock:
            async with aiosqlite.connect(self.db_path) as db:
                cursor = await db.execute(
                    "SELECT url, depth FROM queue WHERE state = 'pending' LIMIT ?", (limit,)
                )
                rows = await cursor.fetchall()
                urls = [{"url": row[0], "depth": row[1]} for row in rows]
                
                for item in urls:
                    await db.execute("UPDATE queue SET state = 'processing' WHERE url = ?", (item['url'],))
                
                await db.commit()
                return urls

    async def fetch_all_pages(self) -> List[Dict]:
        """Retrieve all indexed pages for search."""
        async with aiosqlite.connect(self.db_path) as db:
            db.row_factory = aiosqlite.Row
            async with db.execute("SELECT * FROM pages") as cursor:
                return [dict(row) for row in await cursor.fetchall()]
```

```python
            queue_rows = await queue_res.fetchall()
```

To satisfy the requirements of the `Indexer`, we need to implement `is_visited` (checking both `pages` for completed crawls and `queue` for pending/processing tasks) and `mark_as_pending` (ensuring we don't re-queue existing pages).

### Step-by-Step Thought Process

1.  **Concurrency Strategy**: While `aiosqlite` handles thread safety, `PRAGMA journal_mode=WAL` is already enabled, allowing concurrent readers and a single writer. The `self.write_lock` provided in the boilerplate is appropriate for atomic transactions that span multiple operations (like `save_page_and_links`).
2.  **`is_visited` Implementation**: This needs to be a fast lookup. Since we have a `UNIQUE` constraint on the `url` column in both tables, an `EXISTS` subquery or a simple count check is efficient. I will implement a union-style check.
3.  **`mark_as_pending` Implementation**: This serves as the entry point for discovered URLs. By using `INSERT OR IGNORE`, we maintain the integrity of the queue without needing extra conditional logic, ensuring idempotency.
4.  **Database Efficiency**: Ensure both methods use parametrized queries to prevent SQL injection.
